  • Kurt Fitzner - 2010-11-27

    I've been working on adding parallel processing xz into my disk imaging project (along with parallel gzip and bzip2).  I have to say, I'm disappointed in the .xz container and officially frustrated.  For a library that is an order of magnitude more complex than either zlib or libz2, its container format is incredibly weak.  Gzip set the standard for fantastic compression containers by supporting the ability of library users to add pretty much any arbitrary header data that you wanted to toss in.  Fantastic for adding extra tidbits in for applications that are aware of it, but still keeping compatibility with applications that aren't.  Bzip2 completely dropped the ball with its container format.  It doesn't even have a field telling the compressed block size, let alone custom fields. xz is better than bzip2, marginally, but it still completely lacks the fantastic custom field ability of gzip.  Gzip theoretically has the ability to contain more than just lz77 data.  If folks keep dropping the ball on containers, perhaps it's time to weave bwt and lzma compression engines into the zlib and the gzip container instead of creating new yet inferior containers every go around.  Argh!

  • Lasse Collin - 2010-11-27

    It's very hard to find people to give feedback about a format being designed. So designers are pretty much on their own at guessing what might be needed by the users of the format. Once it is in use, more people look at it and give feedback when changing things is harder. Both situations are obviously frustrating.

    Assuming that you care about making things better and aren't posting just to let off some steam, below is some information why there is no metadata in .xz now. There are also a few questions about what you would need from the metadata data field.

    There was metadata support in some drafts of the format. It supported compression too and could be placed either to the beginning of the file or to the end. There were some severe problems with it though and the design had taken far too long already. So I wanted to simplify things and just get the format done. Thus the metadata field was dropped.

    Metadata support didn't seem too useful. RFC 1952 mentions exactly one use for the .gz extra field, but back then I found that also dictzip used the extra field. dictzip uses it for random-access support and that is supported by the .xz format already (liblzma needs some improvements to make it easier to use). The fact that .bz2 and .lzma didn't have a metadata field and I had never heard anyone complain about it made me more confident that a metadata field isn't very important.

    I did have heard some complaints about gzip's default behavior with metadata. If -no-name isn't used, it writes the original filename and mtime to the header. When comressing in a pipe, filename cannot be known but mtime is set to the current time. So if you compress the same data twice with a pipe using the same gzip binary you get different output every time. It's easy to avoid with a command line option but many don't remember to use it when they might want it.

    I may be wrong but I have understood that storing the filename in the .gz headers was used mostly to allow restoring the original filename on DOS which had 8+3 filename length limit. Using a real archiver to store the file metadata (name, time, permissions, etc.) seems like a better solution nowadays.

    Questions:

    What kind of metadata would you store? Why not store the information in the data stream itself?

    How much metadata would you store at maximum?

    Is it OK to require that the whole metadata is given to the encoder at once (e.g. an array of structures having ID, size, and data)? Or should it be possible to encode metadata just like normal data is encoded so that you don't need to know beforehand how much you will write?

    Does it matter at all if the metadata field is at the beginning of the file or at the end? Note that if it was at the end and seeking is possible and fast, it would be possible to find it quickly.

    Is the .gz-style ID+size+data encoding of the metadata field the best?

    Is there any need to support compression of the metadata?

    I think the simplest way to add metadata support into .xz would be to put it after the Index field and indicate its presence with a bit in the Stream Flags field. It would mean that XZ Utils 5.0.x wouldn't support those files, but considering that the feature wouldn't be used by many people, it probably wouldn't be too bad. But before considering this more, I'll wait for your answers so that I can understand the use cases of metadata fields better. Thanks!

  • Kurt Fitzner - 2010-11-28

    larhzu
    Assuming that you care about making things better and aren't posting just to let off some steam, below is some information why there is no metadata in .xz now. There are also a few questions about what you would need from the metadata data field.

    Well, I am/was frustrated.  It was really difficult to hack metadata into bzip2 in a way that wouldn't break the reference bzip2.exe on decompress.  I ended up having to exploit the fact that there are a spare 4 bits of padding when a single character is compressed in a bzip2 stream.  I then take the first 16 bytes of the data I'm compressing and compress them in independent streams that I concatenate together with their 4 bits of metadata each to get the 64 bits I needed (as a bare minimum).  Not exactly pretty.

    However, that being said, I'm not here just to let off steam.  I would love to work with you on the issue, if indeed there is the possibility of getting metadata into the format.

    larhzu
    What kind of metadata would you store?
    Why not store the information in the data stream itself?
    How much metadata would you store at maximum?

    I wrote a disk imaging program called SelfImage that supports compression with zlib and bzip2.  I'm now adding lzma.  I want metadata so it can add information on the partition that the image is coming from to a disk image.  The original partition size, location of the original partition, the filesystem on it, date the image was taken, and so forth.

    I don't want the data in the stream itself, because I want the user to be able to take an image and, in any old Linux bootable CD or rescue disk, simply decompress it directly into a partition.  I want the metadata to be available in the image in case my imager is the software being used to restore it back to a partition (so it can provide value-added sanity checking on where an image is going and possibly warn the user), but I don't want to force the user to have to use a special tool to restore the image.  Being able to use simple tools any Linux system will have to restore an image is a feature of the images my software makes.

    I don't need much right now - maybe 100 bytes. 

    larhzu
    Is it OK to require that the whole metadata is given to the encoder at once (e.g. an array of structures having ID, size, and data)? Or should it be possible to encode metadata just like normal data is encoded so that you don't need to know beforehand how much you will write?

    For my purposes, requiring that the whole metadata size be known is not an issue.  For the amount that I need (100 bytes) compared to the size of disk images (gigabytes), worrying about saving a few bytes doesn't make sense.

    In general terms, I wouldn't suggest you worry about unknown length metadata or compressed metadata.  I don't think it needs to be that complex.  It's not a multimedia container format where you need multiple independent compressed data streams (which is almost what you're getting to the point of if you go with unknown-length compressed metadata).

    larhzu
    Does it matter at all if the metadata field is at the beginning of the file or at the end? Note that if it was at the end and seeking is possible and fast, it would be possible to find it quickly.

    Since all my work will be on seekable media, location doesn't matter.  I would prefer it be in a known location (ie: guaranteed to be so many bytes from the beginning or end of the file).  I would like to be able to grab the metadata without needing to use the library to decode it's location.

    larhzu
    Is the .gz-style ID+size+data encoding of the metadata field the best? Is there any need to support compression of the metadata?

    It has the advantage of simplicity.  As mentioned above, I don't think there is really any need for metadata compression.  And if there is, the library user can make use of a simple api buffer-to-buffer compression to simply embed a compressed xz as the metadata.  I don't think you need to worry about offering metadata compression.

    larhzu
    I think the simplest way to add metadata support into .xz would be to put it after the Index field and indicate its presence with a bit in the Stream Flags field. It would mean that XZ Utils 5.0.x wouldn't support those files, but considering that the feature wouldn't be used by many people, it probably wouldn't be too bad.

    It's too bad that the xz file format specification requires that all padding bytes be null. Otherwise, it would be simplicity to implement metadata by having a zero-data-length first block and inflating the size of it's block header.  Then you could stick the metadata inside the first block's header (or any block header for that matter).  I don't know if liblzma actually does check to see if the padding is all null - if it doesn't, perhaps you can still do it this way and remove that restriction from the file format spec.  I don't think this would break previous versions.

    In my opinion, checking that every padding byte is null is overcautious.  I think what you really need are two version number fields.  The first is the version of xz file format that the file IS. The second is the minimum version needed to decode the file SAFELY.  That way you can implement flags in future versions that indicate things that don't necesarily make earlier versions unable to decode it.  You can also use padding bytes in the future for whatever you like, with the knowledge that earlier versions of the decoder will ignore them if it doesn't understand them.  Perhaps you can use the version byte for this - top 4 bits are version needed to decode, bottom 4 bits are version the file actually is.

    In any case, after the index is fine for my purposes, and I think most purposes.  There can be arguments for having metadata at the front, though.  If someone is streaming a xz file over a network, it might be nice to have the metadata at the front so it can be extracted and displayed to the user before the whole file is brought over.

  • Kurt Fitzner - 2010-11-28

    I just took a look at lzma_block_header_decode(), and it returns an error if UncompressedSize = 0.  It also, unfortunately, does check to ensure all padding bytes are null, so neither idea will work.  xz is just too robust. :)  It not only checks the bytes it knows its going to have to deal with to make sure they make sense, but it checks the flags and bytes it knows it plans on ignoring.   I would suggest that future versions of xz relax  this.  Have xz trust the version number to tell it when it can't safely handle something, and not return an error if it sees something in a place it intends to ignore anyway.  This will make future expansion a lot less painful.  Anyway, I don't mean to harp on this so enough said.

    Anyway, I had been hoping that xz didn't actually check the padding bytes, so it could be used in a backwards-compatible way.  Since that's not possible, then I can see several ways of marking a block as being metadata.  I see no huge drawback to a block with UncompressedSize = 0 (unless you support compressed metadata blocks), but since this won't work with current xz implementations (it returns an error if UncompressedSize = 0), I don't see why not just add a new block flag to indicate a block is metadata.  You wouldn't even need to update the format version number. Even if you end up relaxing the flag/padding byte checking in new versions, the old versions that don't support the metadata block flag will return an error when they see the flag.  This way, you can still have access to an accurate UncompressedSize value in the case that you do stick lzma in the metadata block's filter chain.  This is valuable information if the metadata block is compressed, so you know how big the metadata is when uncompressed - otherwise if you use UncompressedSize = 0 as the indicator for metadata, then you lose the ability to use it for anything else.

    If I understand the library well enough, then another way to do it is to add a new filter type for metadata.  It would be put as the first filter in the chain for metadata blocks.  On encode it does nothing (data out = data in), on decode it acts as a sink (data out = nil).  That way on decoding a file normally, all metadata blocks get stripped out.  You'd have to use a metadata access function, which would strip the metadata filter out of the filter chain, to decode the data and use it.

    Putting the metadata inside a block means it will be behind two VLIs, so the first piece of metadata won't necessarily be in a fixed location.  This isn't a huge issue.  Yes, I would personally like to see the metadata in a fixed location just to make it easier to extract without necessarily resorting to liblzma, but that might not be important to everyone.  And it could even be made to be at a fixed location by mandating a max-size VLI for cases where the first block is metadata. 0x008080808080808080 is a valid VLI after all.

    I don't see a real issue with allowing LZMA2 in the filter chain either.  I don't see a huge value to compressed metadata myself, but whatever way you use to mark a block as being metadata (a new block flag, a new metadata filter id, UncompressedSize = 0… whatever), you're already going to have to modify xz to support it.  So it seems reasonable that you can also make it so it's not a CPU load bomb.  During normal decompression, just ignore metadata blocks - only decompress them when they are accessed via metadata support function calls.

  • Lasse Collin - 2010-11-29

    To kfitzner:

    Thanks for your answers.

    Currently everything in the .xz headers is meant to somehow describe how the decoder is supposed to decode the compressed data and verify its integrity. If you add a new field that meets these criteria, you would very likely be adding something that the decoder is required to understand to decode the data properly; ignoring the new information wouldn't be an option.

    On the other hand, with a metadata field all new stuff added into that field is meant to be something that is not required to decode the data properly, and thus anything can be put there while keeping old decoders happy. Since there is no metadata field now, adding it will affect old decoders.

    Checking the padding bytes is to check for possible new fields that would be required to decode the file properly. That is, a new version of the format could use that space for something else than pure padding. This is a requirement in the specification and the checks are done also by LZMA SDK. If the padding wasn't checked, someone could use it for their needs and there would be a problem if the official specification would start to use those bytes for some other purpose (that would be much more frustrating than the current situation).

    One could argue that all decoder-must-understand additions should be indicated in Block Flags and optional stuff be allowed in the Header Padding field. Maybe something like that should be considered. If this seems like a good idea, there should be at least something more detailed in the specification so that applications can detect when the metadata was written by them (e.g. an ID followed by arbitrary data). But I hope that a simpler way to store metadata is enough.

    Only the shortest encoded form of variable-length integers are valid. Others are rejected. This is in the specification too. There is a practical reason for this requirement; it's not there to be pedantic or to catch new features.

    About your use case: Would it be dumb to use a .tar containing a small file with the metadata followed by a compressed disk image? You could also include a small README and information about the image in human-readable format. That way people wouldn't need your tool to decode the metadata from .gz/.bz2/.xz headers and your tool wouldn't need so much compression-method specific code either. GNU tar and bsdtar have the -O option to extract a file to stdout, so it would be easy to extract the image (and view README) using only basic tools.

    To ipavlov:

    I think it could be OK to use Uncompressed Size = 0. For example, it doesn't do anything bad for random-access reading.

    I think using a new bit in the Block Flags field is better than using a special Filter ID. Both a new flag bit and a new Filter ID mean that old decoders cannot decode the file. Using a flag bit is a cleaner way in my opinion.

    Here is yet another idea: By far the simplest way to add a metadata field could be to define a new Stream type with a new bit in the Stream Flags field. This would probably be the best way from kfitzner's point of view because decoding (and also encoding) of the metadata field would be quite easy without liblzma, LZMA SDK, or equivalent library.

    The metadata Stream could be put before or after the data Stream. If metadata is after the data Stream, XZ Utils 5.0.x could be used to decompress the data part of the file, although the decompression would end with an error message "Unsupported options" (specifically, it won't be "Compressed data is corrupt").

    The disadvantage of a new Stream type is that the metadata must always be in a separate .xz Stream. This prevents interleaving metadata with the main data while having only one Index (this applies also to my original suggestion). Some rare situations might also not like that they need to deal with multiple Streams. I suppose that these problems are not significant in practice, but it's hard to be sure.
