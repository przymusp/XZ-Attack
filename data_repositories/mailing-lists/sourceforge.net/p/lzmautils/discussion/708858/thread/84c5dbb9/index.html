    • Lasse Collin - 2008-06-12

      You can use lzma_alone_encoder() to create streams compatible with LZMA Utils 4.32.x and the LZMA_Alone tool (lzma.exe) from LZMA SDK. These streams can be uncompressed with lzma_auto_decoder(). However, a future version of liblzma will have new restrictions on supported LZMA_Alone streams (both encoder and decoder):

      - Dictionary size must be 2^n or 2^n + 2^(n-1). Other sizes should be non-existent in practice, so this restriction allows detecting non-LZMA_Alone files better (LZMA_Alone format has no magic bytes).

      - literal_context_bits + literal_pos_bits <= 4. This also means that the maximum value of literal_context_bits will become 4 instead of current 8.

      The format used by lzma_stream_encoder_single() and _multi() is not stable. Actually there are some changes already being made. The APIs will change quite a bit too (there will be just lzma_stream_encoder() and other simplifications). The changes affect also lzma_alone_encoder() and lzma_auto_decoder().

      The format you described in your post is not LZMA_Alone format, since LZMA_Alone format header has also the uncompressed size of the stream (or (uint64_t)(-1) if unknown) after the five-byte properties written by WriteCoderProperties(). For details, look around the WriteCoderProperties() function call in LzmaAlone.cpp in LZMA SDK.

      I can make it possible to decode this kind of custom streams with liblzma if it is needed. Currently there's no separate function to decode the five-byte properties, because so far it has been needed only in the code behind lzma_alone_decoder(). It won't be possible to use lzma_auto_decoder() since there would be no way to distinguish between this custom format and LZMA_Alone streams.

      I know that this is quite messy at the moment. I hope it gets clearer once the new format is stable and supported by 7-Zip, LZMA SDK, and LZMA Utils. Then most people should be able to simply use the new format.

    • Lasse Collin - 2008-06-12

      I'm not very familiar with the .7z format. To my understanding, it stores the properties to headers, that is, separately from the raw LZMA data. Note that .7z files, that contain x86 executables, often use the BCJ2 filter in addition to LZMA. It is very probable that BCJ2 will never be supported by liblzma, because BCJ2 isn't streamable.

      LZMA_Alone has 13-byte header followed by raw LZMA data. The header consist of five-byte properties and eight-byte uncompressed size. So yes, it's just a matter of inserting eight bytes between the properties and the raw LZMA data.

      Note that ZIP allows LZMA with and without "end of stream (EOS)" marker (I usually call it "end of payload marker (EOPM)"). lzma_alone_decoder() expects to find this marker if and only if the uncompressed size is known. So you cannot unconditionally insert the uncompressed size from ZIP headers to the stream going to LZMA_Alone format decoder.

      I suppose that not using EOS in ZIP files is rare, because the specification recommends its use. Things are simpler when EOS is used. Maybe it would be enough to support ZIP files that use EOS.

      I can provide an easier way in liblzma to minimize hacks in applications supporting ZIP files (with and without EOS). However, I'm working on some big changes in liblzma at the moment, so I'm not going add this functionality right now.

    • Igor Pavlov - 2017-10-05

      There are two types of LZMA streams:
      0) Stream with end mark. That end mark adds about 6 bytes to compressed size.
      1) Stream without end mark. You must know exact uncompressed size to decompress such stream.

      LZMA EOS marker is additional data sequence in compressed stream. But only lzma decoder can recognize that EOS marker. It must unpack whole stream before marker.

    • Lasse Collin - 2017-10-05

      The lzma_alone_encoder() API in liblzma always adds the end marker. Currently there is no public API in liblzma to encode without the end marker. If you don't want the end marker, you need to use LZMA SDK or XZ for Java.

  • Lasse Collin - 2017-10-05

    ZIP files support EOS marker. See sections 4.4.4 and 5.8.9 in APPNOTE.TXT.

    I know that there can be other situations where omitting the end marker would be useful so it should be supported in the API. I added it to my to-do list but it won't be implemented soon as I don't get much coding done at the moment, sorry.

    Note that the LZMA_Alone header isn't the same as the one used in ZIP. The ZIP header needs only the first five bytes of the 13-byte LZMA_Alone header.
